Project Context

Global Challenge: Post-pandemic healthcare faces staff shortages, infection risks, and logistical burdens.

The Omani Context

  • Vision 2040: Equitable healthcare for all regions.
  • Geography: Rural families need access to specialists.
  • Current State: 100% Urban / 90% Rural coverage, but specialist access remains a hurdle.

Problem Statement

1. Manual Logistics

Nurses spend ~30% of time on non-clinical transport (meds, linens), reducing patient care time.

2. The "Presence Gap"

Existing telepresence robots are difficult to control manually and lack engagement (just a "screen on wheels").

Typical Nurse Shift Allocation

Project Objectives

  • Autonomous Navigation: Robust SLAM in crowded corridors.
  • Secure Telepresence: Low-latency audio/visual consultation.
  • AI Integration: LLM-driven interaction (Triage/Guidance).
  • System Validation: Pilot scenario (Nurse Station → Patient Bed).

Literature Review: The Research Gap

We analyzed three distinct fields. None provide a holistic solution.

  • Telepresence: Good for diagnosis, bad mobility (Manual).
  • AMRs (Logistics): Great navigation, zero patient interaction.
  • Social Robots: Emotional support, but stationary/limited utility.

Our Solution: Merges these domains using a hybrid architecture.

Part 2: Design & Engineering

Architecture, Subsystems, and Implementation

Requirements Specification

Customer Requirements

  • Autonomous Navigation (Corridors/Rooms).
  • Remote Telepresence (Dr. to Patient).
  • Infection Control (Minimized contact).
  • User Friendly (Non-technical staff).

Engineering Constraints

  • Budget: Max $2,500 USD.
  • Timeline: 2 Semesters.
  • Environment: Standard Hospital Wi-Fi.
  • Safety: ISO 13482 Compliance standards.

Design Alternatives

Alternative Description Pros Cons
A: Commercial Pre-built platform (e.g., TurtleBot 4) High reliability, Fast setup High Cost, Low Customization
B: Custom-Built DIY Frame & Electronics Low Cost, High Learning High Risk, Long Dev Time
C: Hybrid (Selected) Commercial Base + Custom AI Layer Balanced Reliability/Dev Time Integration Complexity

System Selection: AHP Method

Analytic Hierarchy Process used to select the optimal platform.

Criteria Weights

  • Reliability: 47% (Crucial for Hospital)
  • Dev Time: 30%
  • Cost: 15%
  • Customizability: 8%
Winner: Hybrid Platform
Score: 6.76
Allows us to focus on Intelligence innovation while ensuring mobility safety.

Interface

UI
  • Android Tablet App
  • Web Dashboard (Flask)
  • Voice Command Integration

Intelligence Layer: Gemini Integration

We selected Google Gemini 3.0 Flash via AHP for Human-Robot Interaction.

  • Latency: Ultra-fast response suitable for conversation.
  • Context: Large window (1M tokens) for hospital protocols.
  • Cost: Highly efficient API usage.
Function: Translates patient voice queries ("Where is X?") into navigation goals or reassuring dialogue.
